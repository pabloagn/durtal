# Storage

All images — book covers, author photos, media uploads, and import files — are stored in AWS S3. The bucket is organized using a medallion architecture (bronze/silver/gold) that separates raw data from production-ready assets.

---

## S3 Configuration

| Setting | Value |
|---|---|
| Bucket | `durtal` (from `S3_BUCKET` env var) |
| Region | `eu-central-1` (from `AWS_REGION` env var) |
| Client | AWS SDK v3 (`@aws-sdk/client-s3`) |
| Auth | IAM user with read/write policy scoped to the bucket |

The S3 client is initialized as a singleton in `src/lib/s3/client.ts`.

---

## Medallion Architecture

```
S3 Bucket: durtal/
|
+-- bronze/          Raw, unprocessed uploads
|   +-- imports/     Raw CSV/Excel import files
|   +-- covers/      Original cover images from external APIs
|   +-- uploads/     Raw user uploads
|   +-- media/       Raw media uploads (pre-processing)
|
+-- silver/          Validated and parsed data
|   +-- imports/     Parsed JSON, conflict reports, error logs
|   +-- covers/      Validated cover images
|
+-- gold/            Production-ready assets
    +-- covers/      Processed covers (WebP, resized)
    +-- media/       Processed media (WebP, resized, thumbnailed)
    +-- exports/     Generated export files
```

Data always flows in one direction: **bronze -> silver -> gold**. The UI only reads from `gold/`. Raw data in `bronze/` is retained for auditability and reprocessing.

## Key Generation

S3 keys are generated by deterministic functions in `src/lib/s3/keys.ts`:

### Imports

```
bronze/imports/{importId}/{filename}
silver/imports/{importId}/parsed.json
silver/imports/{importId}/conflicts.json
silver/imports/{importId}/errors.json
```

### Covers

```
bronze/covers/{editionId}/original.{ext}
silver/covers/{editionId}/validated.{ext}
gold/covers/{editionId}/cover.webp
gold/covers/{editionId}/thumb.webp
```

### Media

```
bronze/media/{entityType}/{entityId}/{fileId}.{ext}
gold/media/{entityType}/{entityId}/{mediaType}/{fileId}.webp
gold/media/{entityType}/{entityId}/{mediaType}/{fileId}_thumb.webp
```

Where:
- `entityType`: `work` or `author`
- `entityId`: UUID of the work or author
- `mediaType`: `poster`, `background`, or `gallery`
- `fileId`: Generated UUID for the upload

### Exports

```
gold/exports/{exportId}/library_export.csv
```

## Cover Image Pipeline

Triggered when creating or updating an edition with a `coverSourceUrl`.

```
External URL (Google Books, Open Library)
       |
       v
  Download image
       |
       v
  Resize with Sharp:
    - Cover:     400 x 600 max, WebP
    - Thumbnail: 200 x 300 max, WebP
       |
       v
  Upload to S3:
    - gold/covers/{editionId}/cover.webp
    - gold/covers/{editionId}/thumb.webp
       |
       v
  Update edition record:
    - coverS3Key = "gold/covers/{editionId}/cover.webp"
    - thumbnailS3Key = "gold/covers/{editionId}/thumb.webp"
```

Implementation: `processAndUploadCover()` in `src/lib/s3/covers.ts`. Uses Sharp with dynamic import (native module compatibility).

## Media Upload Pipeline

Two-phase process for user-uploaded images (posters, backgrounds, gallery images).

### Phase 1: Client Upload to Bronze

```
Client                          API                              S3
  |                              |                               |
  |-- POST /api/media/process -->|                               |
  |   action: "presign"         |                               |
  |   entityType, entityId       |                               |
  |   filename, contentType      |                               |
  |                              |                               |
  |<-- { url, bronzeKey, fileId }|                               |
  |                              |                               |
  |-- PUT {url} (raw bytes) ----------------------------------->|
  |   Content-Type: image/jpeg   |                        bronze/|
  |                              |                               |
```

### Phase 2: Server Processing to Gold

```
Client                          API                              S3
  |                              |                               |
  |-- POST /api/media/process -->|                               |
  |   action: "process"         |                               |
  |   entityType, entityId       |                               |
  |   mediaType, fileId          |-- GetObject bronze/ -------->|
  |   bronzeKey                  |<-- raw buffer ----------------|
  |                              |                               |
  |                              |-- Sharp resize + WebP         |
  |                              |                               |
  |                              |-- PutObject gold/ ---------->|
  |                              |   full image + thumbnail      |
  |                              |                               |
  |                              |-- INSERT media record         |
  |                              |                               |
  |<-- { media: {...} }          |                               |
```

### Processing Dimensions

Maximum dimensions by media type (aspect ratio preserved):

| Type | Max Width | Max Height |
|---|---|---|
| Poster | 800 | 1200 |
| Background | 1920 | 1080 |
| Gallery | 1600 | 1600 |
| Thumbnail (all) | 400 | 400 |

Output format: WebP for all processed images.

## Pre-Signed URLs

All S3 access uses pre-signed URLs. Credentials are never exposed to the client.

### Upload URLs

Generated via `getPresignedUploadUrl()`. Default expiry: 1 hour. The client PUTs raw bytes directly to S3.

### Read URLs

Generated via `getPresignedReadUrl()`. Default expiry: 1 hour. The `GET /api/s3/read` endpoint redirects to a fresh pre-signed URL.

## Data Flow: Single Book Add

For single book additions (the common case), the flow is simpler — no medallion layers:

```
1. User enters ISBN → API route fetches from Google Books / Open Library
2. Metadata returned to client for preview
3. User confirms → Server action:
   a. Insert work + edition into Neon
   b. Download cover → process with Sharp → upload to S3 gold/covers/
   c. Create instances for assigned locations
4. Redirect to book detail page
```

No bronze or silver layers for single additions — those are only for bulk operations and cover processing.

---

## Data Flow: Bulk Import

Bulk imports (Goodreads CSV, Calibre export, custom CSV) flow through all three medallion layers:

```
1. User uploads CSV → S3 bronze/imports/{id}/raw_file.csv
2. Parse job reads from bronze, validates/cleans each record:
   - Normalizes ISBNs (strip hyphens, validate checksum)
   - Resolves authors (deduplication against existing)
   - Maps Goodreads shelves → Durtal locations/collections
   - Flags duplicates against existing catalogue
3. Parsed output written to S3 silver/imports/{id}/parsed.json
4. User reviews parsed records in UI (preview table with conflict resolution)
5. On user confirmation, approved records are committed:
   - Works/editions → Neon
   - Instances → Neon
   - Cover images fetched and processed through cover pipeline
6. Import status updated to 'completed' in imports table
```

Each import job gets a UUID. All artifacts are namespaced under that UUID in S3 for traceability.
